> **依据文档**：`LamMap_产品级可售文档_完整版.docx`（产品定位、模块、流程、P0/P1、扩展方向）。

## 一、产品目标（文档 §一、二、三）

- **定位**：询盘 → 解析 → 报价 → 模板/附件 → 可外发内容（邮件/附件），提效、降重复劳动。  
- **主流程（7 步）**：录入询盘 → AI 解析 → 匹配产品 → 生成报价 → 选模板出邮件 → 附件打包 ZIP → 发送客户。  
- **已实现模块（与文档对应）**：JWT 与受保护路由、产品库、报价 CRUD、`/quotes/parse`、同步/异步报价生成 Job、模板、附件上传与 AI 建议、单附件生成 Job、ZIP Job、Dashboard 等（路由见 `backend/main.go`）。

**注意事项（上下文）**

- 前端核心页为「新建报价」等，与文档步骤 1～6 强相关；改交互时优先保证**主链不断**（解析 → 生成 → 模板 → 附件/ZIP）。  
- 文档中的「可扩展方向」（团队、CRM、跟进邮件、报表、多语言）**不在本任务 1 范围**，单独立项时再拆任务。

## 二、上线前 P0：必须修复（文档 §四 ↔ 代码对照）

与可售文档 **§四、必须修复** 五条一一对应，按优先级实施并验收。


| #   | 可售文档要求              | 当前实现要点（截至本任务编写时）                                                                                                | 任务要求                                                                                                                                 |
| --- | ------------------- | --------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------ |
| 1   | 密码重置安全（token + 过期）  | `handlers/password.go` 中 `ResetPassword` 仅需 `email` + `newPassword`，**无 token**；`ForgotPassword` 未发邮件、未落库 token | 增加 `password_reset_tokens`（或等价表），`forgot` 生成带过期 token，`reset` 必须校验 token；或 **P0 临时方案**：关闭 `reset-password` 路由 + 前端隐藏忘记密码（见 `需求1.md`） |
| 2   | 所有数据接口 `user_id` 校验 | 报价/产品/模板/附件及各 Job 的 `Get`* 多已比对 `job.UserID` / `quote.UserID`；**仍需全量审计**                                        | 对 `List/Get/Put/Delete` 及所有 `:id`、Job 查询统一 `WHERE user_id = ?`；重点自查 **AI 相关只读接口**是否泄露他人上下文（若未来带 quoteId）                             |
| 3   | 文件上传安全（MIME + 白名单）  | `handlers/upload.go`：**扩展名白名单 + 10MB**，**未做 Content-Type / 魔数校验**；响应里仍返回原始 `filename`                           | 增加 MIME/魔数校验；禁止危险类型；落盘名保持 UUID；必要时限制 `Content-Disposition` 相关风险                                                                      |
| 4   | 异步任务状态机完善           | 已有 `generate_jobs`、`attachment_generate_jobs`、`attachment_zip_jobs` 等                                           | 统一状态 `pending/processing/success/failed`；**processing 在服务重启后可恢复或标记失败**；超时与 `retry_count` 策略与前端轮询上限一致（见 `需求1.md` §4）                  |
| 5   | AI 输出 JSON 结构化限制    | 解析/生成等多处依赖模型输出                                                                                                  | 固定结构 + 后端 `json.Unmarshal` 后校验/归一；禁止仅「尽量 JSON」；大 TEXT 字段建议带 `version`（`需求1.md` §5、§6）                                                |


**注意事项（上下文）**

- **不要只改前端**：P0 以 `backend/handlers`、`backend/models`、`middleware` 为准；密码与权限问题前端隐藏只能算**临时降级**。  
- `main.go` 中 `/api/auth/forgot-password`、`/api/auth/reset-password` **无 JWT**，任何改动需兼容现有前端调用方式或同步改前端。  
- 静态目录 `r.Static("/uploads", "./uploads")` 与上传逻辑配套；安全策略要考虑**直链访问**与文件类型一致性。

## 三、体验优化 P1（文档 §五）

1. 自动保存报价草稿
2. 附件状态可视化（生成中/失败/可下载）
3. 报价编号规则化（如 `Q20260410-0001`）
4. 产品匹配评分优化（加权，缺失字段不硬匹配）
5. 模板分类（系统 / 用户 / AI）

**注意事项（上下文）**

- 附件链路已区分「有 url / 无 url / ZIP 只打有 url 且勾选」；P1 的 UI 要与 `AttachmentPack.vue`、`quote` store 里已有轮询逻辑对齐，避免重复轮询或状态不一致。  
- 报价号若加唯一规则，需考虑 SQLite 下并发与迁移；与 `quotes.quote_number` 现有约束一起设计。

## 四、验收清单（建议按此勾）

**P0**

- 重置密码：token 方案已上线，或接口已关闭且前端无入口  
- 抽样接口：无法用他人 `id` 读改删数据  
- 上传：伪造扩展名/异常 MIME 被拒绝  
- Job：杀进程重启后无永久卡死 `processing`；前端有轮询上限提示  
- AI：解析/生成失败时有明确错误信息，且落库结构可版本演进

**P1**

- 新建报价有草稿自动保存或等价防丢方案  
- 附件列表状态与按钮（生成/预览/重试）一致  
- 报价编号可读且唯一  
- 匹配结果排序可解释（权重或规则文档化）  
- 模板可区分来源，避免升级覆盖用户内容
