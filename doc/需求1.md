3. **异步任务可靠性不足**
4. **附件/文件处理风险**

# 一、必须优先修（建议你第一批就做）

## 1）**重置密码接口现在风险非常高**

你文档里写的是：

- `/api/auth/forgot-password`：MVP 不发邮件
- `/api/auth/reset-password`：**直接按 email 重置**

这在真实环境里基本等于：

> 只要知道别人邮箱，就能改别人密码。

### 必修方案（简单版）

不要先做复杂邮件系统，**最少改成下面这样**：

### 后端简单安全方案

- `forgot-password`
  - 生成一个 `reset_token`
  - 存 DB（带过期时间，比如 15 分钟）
- `reset-password`
  - 必须提交：
    - `email`
    - `token`
    - `newPassword`

### 数据表建议新增

```
password_reset_tokens
- id
- email
- token
- expired_at
- used
- created_at
```

### 如果你现在只想“内部使用，不公开注册”

那也至少要：

- 暂时**关闭 reset-password 接口**
- 前端隐藏“忘记密码”
- 管理员手工改密码

**这是最该先修的点，没有之一。**

## 2）**所有资源必须做“用户归属校验”**

你文档里很多表都有：

- `user_id`
- products
- quotes
- templates
- attachments
- jobs

这很好，但真正危险的地方在于：

> **接口有没有每次都检查“这条数据是不是当前登录用户的”？**

如果没有，就会出现这种问题：

- 我知道 `quoteId=12`
- 直接请求 `/api/quotes/12`
- 就能看别人报价

### 你必须检查的接口

这些都要强制 `WHERE user_id = currentUserID`

#### 必查

- `GET /api/products/:id`
- `PUT /api/products/:id`
- `DELETE /api/products/:id`
- `GET /api/quotes/:id`
- `PUT /api/quotes/:id`
- `DELETE /api/quotes/:id`
- `POST /api/quotes/:id/duplicate`
- `GET /api/templates/:id`
- `PUT /api/templates/:id`
- `DELETE /api/templates/:id`
- `GET /api/attachments`
- `DELETE /api/attachments/:id`
- 所有 `jobId` 查询接口

### Go / Gorm 正确写法示例

```
db.Where("id = ? AND user_id = ?", id, currentUserID).First(&quote)
```

## 3）**上传文件现在缺“真校验”，不能只看扩展名**

你文档写了：

- 支持 pdf / word / excel / 图片
- 限制 ≤10MB
- 上传到 `/uploads/...`

这还不够，因为如果你只是检查文件后缀，别人可以上传：

- `xxx.pdf.exe`
- 伪装图片
- 恶意 HTML / SVG
- 超大异常文件

### 必修点

### 后端必须做：

- 校验文件大小（你已有）
- **校验 MIME Type**
- **限制允许扩展名**
- **文件名不能直接用用户原名**
- 保存时用 UUID（你文档里看起来已经有这方向）
- 禁止上传这些：
  - `.exe`
  - `.bat`
  - `.cmd`
  - `.js`
  - `.html`
  - `.svg`（建议先禁）
  - `.sh`

### 建议允许的最小白名单

```
.pdf
.doc
.docx
.xls
.xlsx
.jpg
.jpeg
.png
.webp
```

### 特别注意

你现在有“AI 生成附件文件”功能，意味着后端也会生成可访问文件。  
**这部分尤其不能直接生成可执行/可注入内容。**

## 4）**异步 Job 必须补“失败可恢复”和“状态机”**

你现在有 3 类异步任务：

- `generate_jobs`
- `attachment_generate_jobs`
- `attachment_zip_jobs`

这是对的，但目前从文档看，**像是“能跑就行”的任务表**，还缺关键可靠性。

如果不补，你以后会遇到：

- 前端一直转圈
- Job 卡死在 `processing`
- 服务重启后任务丢状态
- AI 超时后无法重试

---

### 必修：统一状态机

每个 job 统一只允许这些状态：

```
pending
processing
success
failed
```

可选再加

```sql
retry_count INTEGER DEFAULT 0
started_at DATETIME
finished_at DATETIME
```

### 必修：前端轮询必须有超时策略

比如：

- 每 2 秒轮询一次
- 最多 60 次（2 分钟）
- 超时后提示：
  - “任务仍在处理中，请稍后在历史任务中查看”

### 必修：服务重启恢复

启动时把这些任务修正：

```sql
status = processing
```

统一改回：

```sql
status = failed
error_msg = "service restarted"
```

否则前端永远等不到结果。

## 5）**AI 输出必须“强制结构化”，不能完全信模型**

你项目最核心的是：

- 

解析询盘  

- 

生成报价  

- 

生成模板  

- 

生成询盘示例   

但 AI 最大的问题不是“会不会生成”，而是：

> **生成得不稳定、字段飘、格式飘、语言飘。**

如果你不控，后端会越来越脏。

---

### 必修做法：所有 AI 输出都要“固定 JSON Schema”

例如 `/api/quotes/parse`

不要只让模型“尽量输出 JSON”，而要：

## 后端定义固定结构

```go
type ParsedParams struct {
    ProductName   string   `json:"productName"`
    Material      string   `json:"material"`
    Size          string   `json:"size"`
    Color         string   `json:"color"`
    Quantity      int      `json:"quantity"`
    Budget        string   `json:"budget"`
    DeliveryDate  string   `json:"deliveryDate"`
    Unconfirmed   []string `json:"unconfirmed"`
}
```

## AI 提示词要求

- 只能输出 JSON
- 不能输出 markdown
- 不能解释
- 缺失字段填空字符串或空数组

### 后端再做二次修正

例如：

- 数量不是数字 → 转换/置 0
- 颜色太长 → 截断
- 空字段统一

### 你现在最容易翻车的点

- `parsed_params TEXT`
- `reply_versions TEXT`
- `confirmation_list TEXT`
- `attachment_list TEXT`
- `template_meta TEXT`
- `rendered_contents TEXT`

这些如果全靠 AI 自由发挥，后面会非常难维护。

## 6）**现在很多字段“用 TEXT 存 JSON”，短期快，长期会乱**

你现在 `quotes` 表里很多字段都像：

```sql

- 不好筛选
- 不好统计

### 简单版建议（不用一次全重构）

先不要求全拆表，你可以先做这 3 件事：

### 1）统一 JSON 格式版本

这样以后字段结构变了还能兼容老数据。

例如：

- `customer_name`
- `country`
- `currency`
- `status`
- `total_amount`
- `lead_time`

你已经做了一部分，这很好。继续坚持这个原则。

### 3）附件列表未来建议拆表

特别是：

```
attachmentList

你现在既有：

- 

上传附件  

- 

AI 建议附件名  

- 

AI 生成后回填 url  

这个逻辑已经很“业务化”了，**后面一定建议拆成 quote_attachments 表**，不然前端和后端都容易绕晕。

## 7）**报价单号 quote_number 必须保证唯一且可读**

你现在文档里写了：

```sql
quote_number TEXT UNIQUE
```

但没看到生成规则。

### 不建议

- 用纯随机 UUID 当报价号（太丑）
- 用时间戳秒级（并发可能撞）

### 推荐简单规则

```
Q20260409-0001
Q20260409-0002
```

### 实现建议

- 

每天一个序号  

- 

插入时生成  

- 

冲突重试一次  

这样：

- 

业务上可读  

- 

客户沟通方便  

- 

后续查单方便

## 8）**产品匹配逻辑太“轻”，会导致推荐看起来像乱推**

你现在有：

- `/api/products/match`  
- 

按 `productName/material/size/color/model` 推荐最多 10 条   

这个方向是对的，但如果只是简单 SQL `LIKE`，效果会很差。

---

### 简单可落地优化（不用上向量）

给每个字段加权打分：

```
name 命中 +5
material 命中 +3
size 命中 +3
color 命中 +2
model 命中 +4
```

然后按总分排序。

### 还要补一个逻辑

如果 AI 解析里某字段缺失，不要硬匹配。  

否则用户只说“我要定制马克杯”，你去按 size/color/model 乱筛，结果会很奇怪。

---

## 9）**模板库必须加“默认模板”和“用户模板”区分**

你现在模板库支持：

- 

quotation / email / chat / confirmation  

- 

zh / en  

- 

AI 生成模板并保存   

这很实用，但缺一个关键点：

> 用户到底是在“改系统模板”，还是“存自己的模板”？

### 建议最少加两个字段

```sql
is_default BOOLEAN
is_system BOOLEAN
```

或者更简单：

```sql
source TEXT -- system / user / ai
```

### 为什么必须加

否则你后面会遇到：

- 系统升级后模板覆盖用户内容
- 用户删掉默认模板
- AI 生成模板和手工模板混在一起很乱

# 三、前端层面最该补的（很多项目死在这里）

---

## 10）**新建报价页必须有“自动保存草稿”**

你的核心页面一定是“新建报价”。

这种页面字段多、AI 多、附件多，如果没有自动保存，用户体验会很差。

### 最少做法

前端每 20~30 秒自动保存一次：

- 仅当内容变化时
- 保存为草稿 status = draft

### 建议保存内容

- 客户信息
- 解析结果
- 选中的产品
- 回复内容
- 附件列表
- 模板选择

这会大幅降低“白干了”的概率。

---

## 11）**异步任务前端必须有统一“任务提示框”**

你现在有多个 AI / ZIP 任务，用户如果看不到状态，就会以为“坏了”。

### 建议统一组件

做一个全局任务状态组件，支持：

- 处理中
- 成功
- 失败
- 查看详情
- 重试

### 最少先接入这些

- 报价生成 Job
- 单附件生成 Job
- ZIP 打包 Job

不然体验会很碎。

---

## 12）**附件区现在最容易让用户困惑**

你文档里这个设计其实很聪明：

- 上传附件：有 `url`
- AI 建议附件名：无 `url`
- 可触发单附件生成后回填 `url`
- ZIP 只打包“勾选且有 url”的附件

但是这个逻辑对用户来说不够直觉。

---

### 前端必须补的显示状态

每个附件条目建议明确显示：

- **已上传**
- **AI 建议（未生成）**
- **生成中**
- **生成失败**
- **可下载**

### 并且按钮要按状态切换

例如：


| 状态       | 按钮           |
| -------- | ------------ |
| AI 建议未生成 | 生成           |
| 生成中      | 禁用 + loading |
| 已生成      | 预览 / 下载      |
| 失败       | 重试           |


否则用户会不知道为什么“灰色不可预览”。
